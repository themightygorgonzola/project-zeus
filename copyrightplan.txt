Plan: Copyright Compliance & Open 5e Integration
TL;DR: All hardcoded game data is SRD-compliant. The real risk is AI-generated content — there are zero copyright guardrails in prompts, and zero content validation in sanitization. Fix via prompt guardrails, a content blocklist, optional Open 5e API integration, and proper licensing.

Steps
Phase 1: Prompt-Level Guardrails (highest impact, least code)

Add copyright guidance to narrative system prompt — gm-context.ts → buildNarrativeSystemPrompt()
Add copyright guidance to state extraction prompt — gm-context.ts → buildStateExtractionPrompt() (parallel with 1)
Add copyright guidance to world enrichment prompt — world-enrichment.ts → systemPrompt() (parallel with 1, 2)

Phase 2: Content Validation Layer (defense-in-depth, catches what prompts miss)

Create copyright blocklist module — new file site/src/lib/game/data/copyright-blocklist.ts. Export a Set of ~50-80 trademarked/copyrighted terms and a containsCopyrightedTerm(text) checker. Covers: WotC trademarked creatures (Beholder, Mind Flayer, Displacer Beast, Gith, etc.), iconic NPCs (Strahd, Drizzt, Acererak, Vecna, Elminster), Forgotten Realms locations (Waterdeep, Baldur's Gate, Neverwinter, Candlekeep, Ravenloft), published module names.
Add content validation to sanitizeStateChanges() — adventure-turn.ts: check itemsGained names, npcsAdded names, encounterStarted creature names against blocklist; auto-rename and log (depends on 4)
Add content validation to world enrichment parsing — world-enrichment.ts → parseEnrichmentResponse(): check NPC names and quest text (depends on 4, parallel with 5)

Phase 3: Open 5e API Integration (optional enhancement)

Create Open 5e API client — new file site/src/lib/server/open5e.ts with typed fetch wrappers, document__slug=5esrd filtering, caching layer
Use Open 5e for creature stat block enrichment — creature-templates.ts: check if creature name matches SRD monster before falling back to procedural generation (depends on 7)
Use Open 5e for magic item validation — cross-reference AI-generated magic items against SRD list (depends on 7)

Phase 4: Attribution

Add LICENSE.md or LEGAL.md to repo root with CC-BY-4.0 SRD attribution
