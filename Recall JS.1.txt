================================================================================
  GROUP 1 — FOUC PREVENTION & THEME FLASH FIX
  Purpose: Apply the user's saved theme BEFORE the first browser paint so
           there is never a flash of wrong-theme or unstyled content (FOUC).
  Files modified: site/src/app.html
================================================================================

CHANGE 1.1 — Add visibility:hidden guard on <html> until theme is applied
  File: app.html
  Location: Inside <head>, immediately before the blocking theme script.

  Add this <style> tag as the FIRST thing after the <link> tags in <head>:

      <style>html:not(.fouc-ready) { visibility: hidden; }</style>

  Explanation:
    This CSS hides the entire page until the class "fouc-ready" is added to
    <html>. The hidden state lasts only microseconds — the blocking IIFE
    below adds the class synchronously. The result: the user never sees
    unstyled content, a white flash, or the wrong theme on page load or
    navigation. This is a pure CSS guard; it does not rely on JavaScript
    executing — if JS is disabled the page will remain invisible, which is
    acceptable because this app requires JS.

CHANGE 1.2 — Add blocking IIFE script to restore saved theme before first paint
  File: app.html
  Location: Inside <head>, immediately after the visibility:hidden <style>.

  Add this <script> block (must be a plain IIFE — no type="module", no
  async/defer — so browsers execute it synchronously before rendering):

      <script>
          (function () {
              var VALID = [
                  'arcane-celestial',
                  'dark-academia',
                  'void-astral',
                  'silver-scroll',
                  'forge-iron',
                  'morning-mist',
                  'midnight-court',
                  'clean-slate',
                  'blood-oath',
                  'vellum-quill',
                  'sakura-court',
                  'frozen-spire'
              ];
              try {
                  var saved = localStorage.getItem('nornyx-theme');
                  if (saved && VALID.indexOf(saved) !== -1) {
                      document.documentElement.dataset.theme = saved;
                  }
              } catch (e) {
                  // localStorage unavailable (private browsing, security policy)
                  // Fall through — default theme is used silently.
              }
              // CRITICAL: classList.add must be OUTSIDE the try/catch block.
              // If localStorage throws, we must still reveal the page —
              // otherwise the page stays invisible forever.
              document.documentElement.classList.add('fouc-ready');
          })();
      </script>

  How it works:
    1. The IIFE runs synchronously before the browser paints anything.
    2. It validates the saved theme string against the VALID list —
       never trusting arbitrary localStorage content.
    3. It sets data-theme on <html>, which activates the matching
       CSS [data-theme='...'] block (defined in app.css).
    4. It adds the class "fouc-ready" to <html> (outside try/catch),
       which satisfies the html:not(.fouc-ready) guard and makes the
       page visible.
    5. On first visit (no saved theme), no data-theme is set and the
       CSS :root defaults (arcane-celestial) are used.

CHANGE 1.3 — Complete app.html head structure with all font families
  File: app.html
  Location: <head>, before the FOUC script.

  The <link> tag below loads all 14 font families used across all 12 themes

CHANGE 4.1 — Lock html overflow as well as body
  File: routes/adventures/[id]/+page.svelte

  ORIGINAL CODE (in an $effect that locks scrolling when the page mounts):

      document.body.style.overflow = 'hidden';
      return () => { document.body.style.overflow = ''; };

  CHANGED TO:

      document.documentElement.style.overflow = 'hidden';
      document.body.style.overflow = 'hidden';
      return () => {
          document.documentElement.style.overflow = '';
          document.body.style.overflow = '';
      };

  Why: Locking only <body> leaves <html> scrollable in some browsers,
  causing a visible second scrollbar on the main page. Locking both
  elements eliminates it completely.

CHANGE 4.2 — Parametrize scrollChatToBottom + call with 'instant' on mount
  File: routes/adventures/[id]/+page.svelte

  ORIGINAL function signature:

      function scrollChatToBottom() {

  CHANGED TO:

      function scrollChatToBottom(behavior: ScrollBehavior = 'smooth') {

  And wherever the function performs the scroll, pass behavior through:

      el.scrollTo({ top: el.scrollHeight, behavior });

  In onMount, change the call from:

      scrollChatToBottom();

  TO:

      scrollChatToBottom('instant');

  Why: On initial page load, animating the scroll to the bottom of a
  potentially long transcript looks unpolished and causes layout shift.
  Using 'instant' on mount snaps to the bottom immediately. Subsequent
  scrolls (new messages arriving) continue to use 'smooth' by default.

CHANGE 4.3 — Initialize partyCharacters from server data synchronously
  File: routes/adventures/[id]/+page.svelte

  ORIGINAL declaration:

      let partyCharacters = $state<Record<string, PlayerCharacter>>({});

  CHANGED TO:

      let partyCharacters = $state<Record<string, PlayerCharacter>>(
          Object.fromEntries(
              (((data.gameState as { characters?: PlayerCharacter[] } | undefined)
                  ?.characters ?? []) as PlayerCharacter[])
                  .map((character) => [character.userId, character])

CHANGE 4.4 — Remove inter-panel shadow darkening
  File: routes/adventures/[id]/+page.svelte

  The .adventure-screen rule is the primary layout container for the entire
  active adventure page. It already existed with a full CSS grid layout;
  the property --shadow: none was ADDED as the final line inside that existing
  rule. An implementer must write (or preserve) the complete block:

      .adventure-screen {
          height: calc(100dvh - 3.5rem);
          display: grid;
          grid-template-columns: 240px minmax(0, 1fr) 460px;
          grid-template-rows: auto 1fr;
          grid-template-areas:
              "header header journal"
              "left   main   journal";
          column-gap: 0.85rem;
          row-gap: 0;
          padding: 0.3rem 0 1.5rem;
          overflow: hidden;
          --shadow: none;     /* ← THE ADDED LINE */
      }

  Layout explained:
    - height: calc(100dvh - 3.5rem) — fills the full viewport minus nav height;
      dvh (dynamic viewport height) avoids the mobile address-bar jump.
    - grid-template-columns: 240px left sidebar | flexible main content | 460px journal
    - grid-template-areas: the header row spans both left + main columns;
      the journal panel spans both rows (tall single column on the right).
    - overflow: hidden — each panel manages its own internal scrolling;
      no page-level scroll occurs inside the adventure screen.

  Why: The default --shadow creates an uneven darkening effect between the
  sidebar, main content area, and journal panel. Overriding it at the
  container level is the minimal surgical fix — it does not affect the
  summary page or any other route that uses GlassPanel.

CHANGE 5.1 — Create /api/adventure/[id]/+server.ts
  This file is created from scratch. Full file content:

      import { json, error } from '@sveltejs/kit';
      import type { RequestHandler } from './$types';
      import { db } from '$server/db/client';
      import {
          adventures,
          adventureMembers,
          adventureState,
          adventureChat,
          adventureTurns
      } from '$server/db/schema';
      import { eq, and } from 'drizzle-orm';

      /**
       * PATCH /api/adventure/[id]
       * Body: { status: 'completed' }
       * Owner-only. Marks an adventure as completed.
       */
      export const PATCH: RequestHandler = async ({ params, locals, request }) => {
          if (!locals.user) error(401, 'Not authenticated');

          const { id: adventureId } = params;

          const [adv] = await db
              .select({ id: adventures.id, ownerId: adventures.ownerId })
              .from(adventures)
              .where(and(eq(adventures.id, adventureId), eq(adventures.ownerId, locals.user.id)))
              .limit(1);

          if (!adv) error(404, 'Adventure not found or not owner');

          const body = (await request.json().catch(() => ({}))) as { status?: string };
          if (body.status !== 'completed') error(400, 'Only status: completed is accepted');

          await db
              .update(adventures)
              .set({ status: 'completed', updatedAt: Date.now() })
              .where(eq(adventures.id, adventureId));

          return json({ ok: true });
      };

      /**
       * DELETE /api/adventure/[id]
       * Owner-only. Permanently deletes an adventure and all its data.
       * Child rows are deleted in FK dependency order.
       */
      export const DELETE: RequestHandler = async ({ params, locals }) => {
          if (!locals.user) error(401, 'Not authenticated');

          const { id: adventureId } = params;

          const [adv] = await db
              .select({ id: adventures.id, ownerId: adventures.ownerId })
              .from(adventures)
              .where(and(eq(adventures.id, adventureId), eq(adventures.ownerId, locals.user.id)))
              .limit(1);

          if (!adv) error(404, 'Adventure not found or not owner');

          // Delete in FK dependency order: children before parent
          await db.delete(adventureChat).where(eq(adventureChat.adventureId, adventureId));
          await db.delete(adventureTurns).where(eq(adventureTurns.adventureId, adventureId));
          await db.delete(adventureState).where(eq(adventureState.adventureId, adventureId));
          await db.delete(adventureMembers).where(eq(adventureMembers.adventureId, adventureId));
          await db.delete(adventures).where(eq(adventures.id, adventureId));

          return json({ ok: true });
      };

  Security notes:
    - Both handlers require locals.user (auth guard) — unauthenticated
      calls get 401 immediately.
    - Both handlers verify ownerId === locals.user.id before touching any
      data — non-owners get 404 (not 403, to avoid leaking existence).
    - PATCH only accepts status='completed' — any other body is rejected
      with 400 to prevent arbitrary status mutation.
    - DELETE order matters — adventureChat first, then adventureTurns,
      then adventureState, then adventureMembers, then adventures. Reversing
      this order will violate foreign key constraints.

CHANGE 8.2 — Create summary/+page.server.ts
  File: routes/adventures/[id]/summary/+page.server.ts  (NEW FILE)

  This file is the server-side load function for the completed adventure
  summary page. Full content:

      import { redirect, error } from '@sveltejs/kit';
      import type { PageServerLoad } from './$types';
      import { db } from '$server/db/client';
      import {
          adventures, adventureMembers, adventureState,
          adventureTurns, adventureChat, users
      } from '$server/db/schema';
      import { eq, and, asc } from 'drizzle-orm';
      import { createInitialGameState, migrateState } from '$lib/game/state';
      import type { TurnRecord, StateChange } from '$lib/game/types';

      export const load: PageServerLoad = async ({ params, locals, url }) => {
          if (!locals.user) {
              redirect(302, `/auth/login?returnTo=${encodeURIComponent(url.pathname + url.search)}`);
          }

          const [adv] = await db
              .select()
              .from(adventures)
              .where(eq(adventures.id, params.id))
              .limit(1);

          if (!adv) error(404, 'Adventure not found');

          // Guard: redirect away if not actually completed
          if (adv.status === 'lobby') redirect(302, `/adventures/${params.id}/lobby`);
          if (adv.status === 'active') redirect(302, `/adventures/${params.id}`);

          const [membership] = await db
              .select()
              .from(adventureMembers)
              .where(and(
                  eq(adventureMembers.adventureId, params.id),
                  eq(adventureMembers.userId, locals.user.id)
              ))
              .limit(1);

          if (!membership) error(403, 'You are not a member of this adventure');

          const members = await db
              .select({
                  userId: adventureMembers.userId,
                  role: adventureMembers.role,
                  username: users.username,
                  avatarUrl: users.avatarUrl
              })
              .from(adventureMembers)
              .innerJoin(users, eq(adventureMembers.userId, users.id))
              .where(eq(adventureMembers.adventureId, params.id));

          const [stateRow] = await db
              .select()
              .from(adventureState)
              .where(eq(adventureState.adventureId, params.id))
              .limit(1);

          let gameState = createInitialGameState(adv.worldSeed ?? '');
          if (stateRow) {
              try { gameState = migrateState(JSON.parse(stateRow.stateJson)); }
              catch { /* corrupted state — fall back to empty */ }
          }

          // Load ALL turns in chronological order (no limit — story is frozen)
          const turnRows = await db
              .select()
              .from(adventureTurns)
              .where(eq(adventureTurns.adventureId, params.id))
              .orderBy(asc(adventureTurns.turnNumber));

          const allTurns: TurnRecord[] = turnRows.map((r) => ({
              id: r.id,
              turnNumber: r.turnNumber,
              actorType: r.actorType as TurnRecord['actorType'],
              actorId: r.actorId,
              action: r.action,
              intent: r.intent as TurnRecord['intent'],
              status: (r.status ?? 'completed') as TurnRecord['status'],
              resolvedActionSummary: r.resolvedSummary ?? '',
              mechanicResults: JSON.parse(r.mechanicsJson) as TurnRecord['mechanicResults'],
              stateChanges: JSON.parse(r.stateChangesJson) as StateChange,
              narrativeText: r.narrativeText,
              timestamp: r.createdAt
          }));

          // Load ALL chat in chronological order (no limit)
          const chatRows = await db
              .select()
              .from(adventureChat)
              .where(eq(adventureChat.adventureId, params.id))
              .orderBy(asc(adventureChat.createdAt));

          const allChat = chatRows.map((r) => ({
              id: r.id,
              userId: r.userId,
              username: r.username,
              text: r.text,
              mentions: JSON.parse(r.mentionsJson) as string[],
              retroInvoked: r.retroInvoked,
              consumedByTurn: r.consumedByTurn,
              createdAt: r.createdAt
          }));

          // ── Compute 14 stats ─────────────────────────────────────────────
          const msPlayed = adv.updatedAt - adv.createdAt;
          const hoursPlayed = Math.floor(msPlayed / 3_600_000);
          const minutesPlayed = Math.floor((msPlayed % 3_600_000) / 60_000);

          const enemyRoles = new Set(['hostile', 'boss']);
          const enemiesDefeated = gameState.npcs.filter(n => !n.alive && enemyRoles.has(n.role)).length;
          const bossesDefeated  = gameState.npcs.filter(n => !n.alive && n.role === 'boss').length;
          const alliesLost      = gameState.npcs.filter(n => !n.alive && (n.role === 'ally' || n.role === 'companion')).length;

          const questsCompleted = gameState.quests.filter(q => q.status === 'completed').length;
          const questsFailed    = gameState.quests.filter(q => q.status === 'failed').length;
          const questsActive    = gameState.quests.filter(q => q.status === 'active' || q.status === 'available').length;

          const locationsVisited = gameState.locations.filter(l => l.visited).length;
          const locationsTotal   = gameState.locations.length;

          const totalXp   = gameState.characters.reduce((s, c) => s + (c.xp   ?? 0), 0);
          const totalGold = gameState.characters.reduce((s, c) => s + (c.gold ?? 0), 0);

          const itemsAcquired = allTurns.reduce((sum, t) => {
              const gained = (t.stateChanges as StateChange)?.itemsGained ?? [];
              return sum + gained.reduce((s, g) => s + (g.item?.quantity ?? 1), 0);
          }, 0);

          const stats = {
              hoursPlayed, minutesPlayed,
              totalTurns: allTurns.length,
              totalMessages: allChat.length,
              enemiesDefeated, bossesDefeated, alliesLost,
              questsCompleted, questsFailed, questsActive,
              locationsVisited, locationsTotal,
              totalXp, totalGold,
              daysElapsed: gameState.clock?.day ?? 1,
              itemsAcquired
          };

          return {
              adventure: adv,
              members,
              gameState,
              stats,
              allTurns,
              allChat,
              currentUserId: locals.user.id
          };
      };

  GROUP 9 — DISPLAY HEADING DESCENDER CLIP FIX
  Purpose: Letters with descenders (g, y, p, q, Q, Y) were visually cut off
           on all large page-title headings. The bottom stroke of those glyphs
           appeared to be sliced by an invisible horizontal line.
  Root cause: Every affected heading uses  -webkit-background-clip: text
              (gradient fill). With this technique the browser paints the
              gradient only inside the element's own CSS box. Line-heights of
              1.1 – 1.15 produce a box that is too tight: descenders extend
              below the line box, land outside the painted region, and become
              invisible because -webkit-text-fill-color: transparent has made
              the real text colour transparent.
  Fix: Add padding-bottom: 0.15em (0.1em for the summary page heading) to
       each affected element. Padding expands the element's box downward,
       giving the background-clip paint region enough room to cover every
       descender. Line-height was also nudged from 1.1/1.15 → 1.2/1.15 to
       add consistent breathing room between wrapped lines.
       The summary page hero-title does NOT use background-clip (plain text
       colour) but received the same line-height and padding-bottom treatment
       to prevent any parent overflow: hidden from clipping descenders.
  Files modified: routes/+page.svelte

  GROUP 19 — ORANGE & LEAD DRAGONBORN + GRID REORDER + TYPE SEPARATORS (Session 6)
  Purpose: Add two new Dragonborn subraces (Orange/Fire and Lead/Poison),
           reorder the entire 2-column subrace grid so fire is at the top and
           poison at the bottom, align damage types across columns, order
           chromatics by colour spectrum and metallics by precious-metal value,
           and add thin CSS divider lines between each damage-type group.
  Files modified: site/src/lib/game/data/races.ts
                  site/src/lib/components/CharacterCreation.svelte
  Session date: April 6, 2026

  New subrace design decisions:
    Orange (Chromatic/Fire) — copyright-clear; no TTRPG publisher has ever
      claimed "Orange Dragon" as a type. Orange is biology's universal warning
      colour (dart frogs, monarch butterflies) but shares fire damage with Red
      for a second chromatic fire entry at the top of the grid. Mechanics are
      identical to Red (15 ft. cone, fire damage + fire resistance).
    Lead (Metallic/Poison) — copyright-clear; no D&D edition, supplement, or
      third-party publisher has published a "Lead Dragon" type. Lead's real-
      world toxicology (accumulative heavy-metal poisoning, cognitive effects,
      "sugar of lead" sweet scent masking danger) makes it narratively distinct
      from Green. Mechanics are identical to Green (15 ft. cone, poison damage
      + poison resistance).

  New grid order:
    Row 1 — Red (Fire/Cone)     | Gold (Fire/Cone)
    Row 2 — Orange (Fire/Cone)  | Brass (Fire/Line)
    ── fire separator ──────────────────────────────
    Row 3 — Blue (Lightning/Line) | Bronze (Lightning/Line)
    ── lightning separator ─────────────────────────
    Row 4 — White (Cold/Cone)   | Silver (Cold/Cone)
    ── cold separator ──────────────────────────────
    Row 5 — Black (Acid/Line)   | Copper (Acid/Line)
    ── acid separator ──────────────────────────────
    Row 6 — Green (Poison/Cone) | Lead (Poison/Cone)

  Ordering rationale:
    Chromatic column — spectrum order (Red → Orange → Blue → White → Black →
      Green). Yellow is skipped (no yellow subrace). White and Black are
      achromatic so fall after the spectral colours before the non-colour Green.
    Metallic column — precious-metal value order (Gold → Brass → Bronze →
      Silver → Copper → Lead). Lead is least valuable and most toxic, naturally
      anchoring the poison row.
    Cross-column type matching is the primary constraint — both entries in a
      row share the same damage type. The spectrum/value ordering is applied
      as a secondary constraint within that requirement.
