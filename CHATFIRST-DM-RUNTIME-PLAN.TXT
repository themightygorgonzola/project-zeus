### Slice 2 — Authoritative Resolvers (Complete)

Implemented four authoritative resolver paths that call real engine functions before AI narration:

**use-item resolver** (step 4 — inventory)
- Matches item name from action text against inventory
- Calls `useConsumable()` from inventory engine
- Produces `MechanicResult` (healing dice) and `StateChange` (HP changes, items lost)
- Falls back to clarification if multiple consumables match ambiguously
- Single consumable used implicitly when no name match but only one option

**rest/wait resolver** (step 4 — rest + clock)
- Detects long vs short rest from natural language
- `shortRest()`: auto-spends up to half available hit dice, advances clock 2 periods
- `longRest()`: heals entire party, restores spell slots, conditions, death saves, advances clock 8 periods
- `wait`: parses "until nightfall/morning" or "N hours", advances clock appropriately
- Uses actual 5-period TIME_CYCLE: dawn → morning → afternoon → dusk → night

**travel resolver** (step 4 — travel)
- Matches destination from action text against connected location names
- Calls `travelBetween()` from travel engine
- Produces location change, clock advance, weather generation, random encounter check
- Falls back to clarification if multiple destinations match
- Falls through to AI if no location name matches (AI may create new locations)

**cast-spell resolver** (step 4 — spellcasting)
- Identifies spell from action text by matching against character's known/prepared/cantrip lists
- Validates via `canCastSpell()`, then calls `castSpell()` on a cloned character
- Produces damage/healing dice, slot usage, concentration tracking
- Falls through to AI if spell can't be identified or can't be cast

**Pipeline wiring** (step 1)
- `resolveTurn()` now dispatches to resolvers via `switch(intent.primaryIntent)`
- `ResolvedTurn` now carries `updatedCharacters?: PlayerCharacter[]` for engine mutations
- `persistResolvedTurnAndState()` applies `updatedCharacters` to state before saving
- Attack, talk, examine, free-narration intents still pass through to AI (no engine resolution yet)

What this means mechanically:

- The game should be **chat-primary**, not button-primary.
- UI should be **reference and inspection**, not the main control surface.
- The turn executor should accept **natural command turns**, not rigid action forms.
- The AI can have **broad discretion** when interpreting intent.
- But for **resource-spending ambiguity** like heals/items, the system should **ask follow-up** instead of silently guessing.
- AI world updates should be **creative but bounded by typed state schemas**.

That is a very specific design, and it changes the executor architecture in an important way:

The AI should **not** be the final rules authority. It should be:
1. an **intent interpreter**,
2. a **bounded world-authoring agent**,
3. and a **narrator**,

while the server-side executor remains the authoritative source of:
- legality,
- costs,
- dice,
- targets,
- conditions,
- inventory changes,
- encounter state,
- quest state,
- travel/rest timing,
- and persistence.

There are a few concrete constraints already visible in the codebase:
- current live turns still go straight from chat text to LLM output in adventure-turn.ts
- current client streaming shows raw AI chunks, which conflicts with structured JSON-style GM responses in [site/src/routes/adventures/[id]/+page.svelte](site/src/routes/adventures/[id]/+page.svelte) and gm-context.ts
- state updates are only partially applied in adventure-turn.ts
- the mechanics engine already exists and is ready to be wired in via combat.ts, spellcasting.ts, rest.ts, inventory.ts, travel.ts, and leveling.ts

So the right next move is not “more engine work.” It is **turn architecture**.

## Plan: Chat-First DM Runtime

Build an authoritative, chat-forward runtime where players act through natural language, the server resolves mechanics and state, and AI interprets and narrates within bounded authority. Based on your decisions, the first playable version should use a **natural-command turn model**, allow **broad AI discretion for general interpretation**, require **follow-up clarification for ambiguous healing/item/resource choices**, and allow **bounded AI world creation** through typed schemas only. The key shift is to replace the current LLM-first turn path with a staged pipeline: intent parse → authoritative resolution → bounded world mutation → narration → realtime state/event broadcast.

**Steps**
1. Replace the current LLM-first turn path in adventure-turn.ts with a staged `executeTurn()` architecture:
   - `parseTurnIntent()`
   - `resolveTurnActions()`
   - `applyResolvedStateChanges()`
   - `narrateResolvedOutcome()`
   - `broadcastTurnEvents()`
   This keeps AI involved, but only inside explicit stages.

2. Introduce a typed `ResolvedTurn` model in types.ts or a new game-runtime file to represent:
   - parsed intent
   - chosen action(s)
   - actor
   - target(s)
   - consumed resources
   - mechanic results
   - authoritative state changes
   - clarification-needed status
   This becomes the canonical handoff between parser, executor, and narrator.

3. Split AI responsibility into three bounded roles:
   - `Intent Interpreter`: converts fuzzy chat into structured candidate intents
   - `Narrator`: converts resolved outcomes into prose
   - `World Builder`: background-only typed proposals for quests/NPCs/locations/buildings
   The first two belong in the live turn loop; the third belongs in Trigger/background jobs via adventure-turn.ts and new task files under trigger.

4. Build a new authoritative intent-resolution layer over the existing game engine:
   - combat actions call `createEncounter()`, `resolveAttack()`, `resolveNpcAttack()`, `resolveEncounter()` from combat.ts
   - spells call `canCastSpell()`, `castSpell()`, `expendSpellSlot()` from spellcasting.ts
   - item/equipment actions call inventory functions in inventory.ts
   - rest/wait actions call short/long rest + clock advance in rest.ts and travel.ts
   - travel/discovery actions call `travelBetween()` and `discoverLocation()` in travel.ts
   - level-up actions call `applyLevelUp()` in leveling.ts

5. Add a clarification policy layer for ambiguous resource use:
   - if one valid spell/item/target exists, resolve automatically
   - if multiple reasonable heals, potions, poisons, or consumables exist, mark the turn as `needsClarification`
   - return a short in-world follow-up prompt instead of consuming anything
   This directly implements your “ask follow-up” preference.

6. Keep “broad AI discretion” but constrain it with a decision rubric:
   - AI may infer missing low-risk intent details like nearest enemy, closest ally, default equipped weapon, or obvious movement path
   - AI may not silently choose among multiple materially different resource expenditures when the player owns several valid options
   - AI may not violate action economy, visibility, spell legality, slot availability, inventory, or encounter state
   Put this rubric into the parser/narrator prompt policy in gm-context.ts and the new executor layer.

   - optional `sceneFactsAdded`
